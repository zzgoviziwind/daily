## 一、大模型基础核心知识

### 1. Attention 机制与 Tokens
- **Attention**：大模型理解上下文的核心机制
- **Tokens 估算**：英文 1 token≈0.75 单词≈4 字符；中文 1 token≈1-2 字符
- **超参数调优**：temperature（创造性）、top_k（采样范围）、top_p（累积概率）

### 2. 提示词工程 4 要素
| 要素 | 说明 |
|------|------|
| 你是谁 | 角色设定（如"资深测试专家"） |
| 你在哪 | 上下文/场景描述 |
| 要干什么 | 具体任务目标 |
| 怎么告诉我 | 输出格式要求 |

### 3. 高级提示词技巧
- **思维链（CoT）**：引导模型逐步推理
- **少样本提示（Few-shot）**：提供示例提升准确率
- **角色催眠**：强化角色设定提升专业度

## 三、MCP 协议（Model Context Protocol）

### MCP 核心价值
- 统一的 AI 应用与外部工具通信协议
- 三种基础能力：**Tools**（工具调用）、**Resources**（资源读取）、**Prompts**（提示词模板）
- 两种传输方式：STDIO（本地进程）、SSE（服务器发送事件）

### FastMCP Server 示例
```python
from mcp.server import FastMCP
mcp = FastMCP("Demo")

@mcp.tool()
def add(a: int, b: int) -> int:
    return a + b

if __name__ == "__main__":
    mcp.run(transport='stdio')
```

## 四、测试自动化核心实践

### 1. 需求条目化 + RAG 生成测试用例
- 将需求文档转化为结构化条目
- 利用 RAG 检索相似历史用例
- 生成覆盖正常/异常/边界场景的测试用例

### 2. RAG+Text2SQL 自动生成接口测试
- 将 API 文档存入向量库
- 通过自然语言查询生成 SQL
- 自动生成接口测试脚本

### 3. Playwright MCP Server（意图驱动 UI 自动化）
- 用自然语言描述测试意图
- MCP 协议连接 Playwright 浏览器工具
- 自动执行点击、输入、验证等操作

## 五、5 种 Agentic 设计模式

| 模式 | 说明 | 测试场景示例 |
|------|------|-------------|
| **反思（Reflexion）** | 自我评估输出并改进 | 测试用例质量迭代优化 |
| **提示链（Chain of Thought）** | 分解复杂问题逐步求解 | 分步执行测试分析 |
| **规划（Planning）** | 创建执行路线图 | 测试任务分解与排期 |
| **并行处理（Parallelization）** | 同时执行独立任务 | 多浏览器并发测试 |
| **路由（Routing）** | 根据任务类型选择处理路径 | 按模块分发测试任务 |

## 六、Query 改写技术

### 三种改写方法
1. **基于大模型改写**：直接用 LLM 生成多样变体
2. **词汇表改写**：基于预定义同义词表规则替换
3. **同义词改写**：LLM 实时生成同义词再组合

### 质量验证指标
- **ROUGE-L**：语义相似度（越高越好）
- **BLEU**：词汇差异度（越低表示变化越多）
- **帕累托最优**：平衡多样性与准确性

## 七、测试左移实践

### 开卡（KO）与验卡（DC）
- **开卡**：开发前评审需求，补充验收条件
- **验卡**：开发后按验收条件初步验收

### 大模型友好度评估维度
| 维度 | 评估要点 |
|------|---------|
| 模块化程度 | 能否拆解为独立原子任务 |
| 遗留系统依赖性 | 是否过度锚定老旧系统 |
| 迭代频率与动态性 | 是否高频更新 |

### 智能体四维分析
- **合理性**：业务逻辑链条完整性
- **可行性**：技术栈匹配与资源估算
- **兼容性**：与现有功能/接口交互
- **可测试性**：边界条件与异常路径

## 八、组织落地实战经验

### 推广三阶段
1. **愚昧之巅**：简化演示，快速获得采纳（60%+）
2. **绝望之谷**：自研平台遇冷（流程刚性/产出不稳/算力紧张）
3. **开悟之坡**：转向 MCP Server 方案

### MCP Server 封装避坑指南
- ✅ 整合相关 API 到一个 Server
- ✅ 命名见名知意（如 `jira_search`）
- ✅ 过滤无语义内容（如 `id=10001`）
- ✅ 错误信息要有意义（用文字描述不用错误码）
- ❌ 不要直接暴露原始 API
- ❌ 不要一次性返回大量日志
